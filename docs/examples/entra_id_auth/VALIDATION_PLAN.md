# Microsoft Entra ID Auth Validation Plan

> Status: Automation Ready
> Scope: External functional validation of Marreta JWT authentication against
> real Microsoft Entra ID access tokens.

---

## 1. Goal

Validate that Marreta can protect routes using JWT access tokens issued by
Microsoft Entra ID.

This validation is intentionally external. It proves Marreta's JWT validation
against a real identity provider, but it does not make Marreta responsible for
creating Azure resources or acquiring tokens.

---

## 2. Non-Goals

This validation does not:

- create Microsoft Entra ID App Registrations
- create client secrets
- grant admin consent
- call Azure SDKs from the test script
- use Azure CLI as a test dependency
- validate Microsoft SDK behavior
- run inside the runtime repo's `cargo test`

Token acquisition is caller-owned.

---

## 3. Boundary

The test receives a token that was already acquired externally.

Accepted acquisition methods:

- Azure SDK
- Azure CLI
- manual script
- CI secret injection
- any OAuth client able to request an Entra ID access token

The validation starts after a JWT access token exists.

---

## 4. Required Azure Setup

The operator must prepare two App Registrations in Microsoft Entra ID.

### API Application

Represents the Marreta API.

Required information:

- Tenant ID
- API application client ID
- Expected issuer
- Expected audience

Recommended audience format:

```text
api://<api-application-client-id>
```

The API app should expose an application permission or app role if authorization
by role is part of the validation.

### Client Application

Represents the caller that obtains tokens.

Required information:

- Tenant ID
- Client application ID
- Client secret or another credential supported by the operator's token
  acquisition flow
- Permission granted to the API application
- Admin consent granted

For client credentials, request the API resource with `/.default`:

```text
api://<api-application-client-id>/.default
```

---

## 5. Required Environment Variables

The validation script should require:

```bash
ENTRA_TENANT_ID=
ENTRA_ISSUER=https://sts.windows.net/<tenant-id>/
ENTRA_AUDIENCE=api://<api-application-client-id>
ENTRA_JWKS_URL=https://login.microsoftonline.com/common/discovery/keys
ENTRA_ACCESS_TOKEN=
```

Optional negative-case inputs:

```bash
ENTRA_BAD_AUDIENCE_TOKEN=
ENTRA_BAD_ISSUER_TOKEN=
ENTRA_EXPIRED_TOKEN=
```

If optional tokens are absent, the script should skip only the corresponding
negative case and report it as skipped.

---

## 6. Marreta Project Shape

The validation project should be minimal.

`app.marreta`:

```marreta
project_name = "entra-id-auth-validation"
project_version = "0.1.0"
```

Protected route:

```marreta
route GET "/secure" take auth
    reply 200, {
        ok: true,
        subject: auth.sub,
        audience: auth.aud,
        issuer: auth.iss
    }
```

Optional role-protected route, only if current Marreta auth supports role/claim
authorization:

```marreta
route GET "/secure/role" take auth
    require auth.roles.includes("marreta.validation") else fail 403, {
        error: "forbidden"
    }
    reply 200, { ok: true }
```

## 7. Runtime Configuration

Configuration for the client-credentials token generated by this validation:

```env
ENTRA_ISSUER=https://sts.windows.net/<tenant-id>/
ENTRA_AUDIENCE=api://<api-application-client-id>
ENTRA_JWKS_URL=https://login.microsoftonline.com/common/discovery/keys
```

The generated `auth jwt entra_id` provider reads these values through `env.*`:

```marreta
auth jwt entra_id {
    issuer: env.ENTRA_ISSUER
    audience: env.ENTRA_AUDIENCE
    jwks_url: env.ENTRA_JWKS_URL
    algorithm: "RS256"
}
```

Client credentials tokens issued for this setup are Entra ID v1 access tokens
(`ver = "1.0"`), whose issuer is `https://sts.windows.net/<tenant-id>/`.
The example configures `jwks_url` explicitly because issuer-based OIDC discovery
from `sts.windows.net` is not the correct metadata path for this token shape.

## 8. Validation Scenarios

### Scenario 1 — Public health route

Request:

```bash
curl http://127.0.0.1:<port>/_health
```

Expected:

- HTTP `200`
- proves the app is running before auth-specific checks

### Scenario 2 — Missing token

Request:

```bash
curl http://127.0.0.1:<port>/secure
```

Expected:

- HTTP `401`
- response body identifies authentication failure

### Scenario 3 — Valid Entra ID access token

Request:

```bash
curl \
  -H "Authorization: Bearer ${ENTRA_ACCESS_TOKEN}" \
  http://127.0.0.1:<port>/secure
```

Expected:

- HTTP `200`
- response includes `ok: true`
- response includes expected issuer
- response includes expected audience
- response includes a non-empty subject or app identity claim

### Scenario 4 — Wrong audience

Only run when `ENTRA_BAD_AUDIENCE_TOKEN` is provided.

Expected:

- HTTP `401`
- proves Marreta rejects a valid Entra token minted for a different resource
  such as Microsoft Graph

### Scenario 5 — Wrong issuer

Only run when `ENTRA_BAD_ISSUER_TOKEN` is provided.

Expected:

- HTTP `401`
- proves Marreta does not accept tokens from another tenant/issuer

### Scenario 6 — Expired token

Only run when `ENTRA_EXPIRED_TOKEN` is provided.

Expected:

- HTTP `401`
- proves token lifetime validation is enforced

### Scenario 7 — Role/claim authorization

Only run if the Marreta auth surface supports this today.

Expected:

- valid token with required role → HTTP `200`
- valid token without required role → HTTP `403`

## 9. Script Behavior

The future `test_entra_id.sh` should:

1. Check required environment variables.
2. If required variables are missing, exit with a clear `SKIP` message.
3. Render or copy `marreta.env` from the provided env variables.
4. Start `marreta serve` on a random/free port.
5. Wait for `/_health`.
6. Run validation scenarios.
7. Stop the Marreta process on exit, including failures.
8. Print a concise summary:

```text
PASS missing token rejected
PASS valid Entra token accepted
SKIP wrong audience token not provided
PASS wrong issuer token rejected
```

The script should not persist tokens to disk.

## 10. Example Token Acquisition Outside The Test

This is documentation only. The validation script should not depend on Node.js.

```javascript
import { ClientSecretCredential } from "@azure/identity";

const credential = new ClientSecretCredential(
  process.env.AZURE_TENANT_ID,
  process.env.AZURE_CLIENT_ID,
  process.env.AZURE_CLIENT_SECRET
);

const token = await credential.getToken(
  `api://${process.env.AZURE_API_CLIENT_ID}/.default`
);

console.log(token.token);
```

Usage:

```bash
export ENTRA_ACCESS_TOKEN="$(node get-token.js)"
./test_entra_id.sh
```

---

## 11. Success Criteria

The validation is successful when:

- Marreta accepts a real Microsoft Entra ID access token for the configured API.
- Marreta rejects missing tokens.
- Marreta rejects tokens with wrong audience when provided.
- Marreta rejects tokens with wrong issuer when provided.
- Marreta exposes authenticated claims to route code consistently with the auth
  spec.
- The validation can be rerun without changing source code.

## 13. Automation Files

Implemented files:

- `app.marreta` — project entrypoint and `auth jwt entra_id` provider.
- `routes/secure.marreta` — protected routes.
- `get_token.mjs` — optional Azure SDK token acquisition.
- `package.json` — Node dependency metadata for `@azure/identity`.
- `test_entra_id.sh` — external functional validation script.
- `marreta.env` — local runtime defaults and documented auth envs for this
  runnable validation project.
- `README.md` — operator-facing instructions.

Automation choices:

- `auth jwt entra_id` uses explicit `jwks_url`.
- `algorithm: "RS256"` is configured because Microsoft Entra ID JWKS entries do
  not include `alg`; Marreta requires an explicit algorithm when a JWK omits it.
- `issuer` and `audience` are read through `env.*` at Marreta startup.
- `/secure` validates the token and returns selected claims.
- `/secure/role` is available but only exercised when
  `ENTRA_TEST_ROLE_CHECK=1`.
- Optional negative cases are skipped unless the corresponding token is
  provided.
