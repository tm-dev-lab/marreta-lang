## 1. Context

This document registers the performance analysis of the `v0.4.0` load test with the integration of the PostgreSQL database for the `/products` and `/orders` endpoints.

The goal is to provide a comprehensive comparison between the purely CPU-bound interpreter execution (baseline) and the realistic execution involving Network/Disk I/O and Connection Pooling overhead.

## 3. Findings & Architectural Suspicions

Após o processamento desta rodada contra o ambiente real com PostgreSQL (`marreta-postgres`), pontuamos três descobertas cruciais que os desenvolvedores externos e a arquitetura precisam avaliar.

### 3.1. O Impacto Direto do I/O
A latência em `products` foi de **0.9ms para 36.9ms (p95)**. Isso valida que o gargalo real de execução não está no AST parse ou no engine do MarretaLang, e sim no tempo de round-trip TCP para o PostgreSQL. Truncar ~4.000 requisições/segundo de escritas no PostgreSQL dentro de contêineres Docker prova que a engine está operando maravilhosamente bem e enfileirando I/O via Tokio corretamente.

### 3.2. A Anomalia do `/orders` vs `/products`
Logicamente, `/orders` é um endpoint muito mais pesado que `/products` no Marreta (validações aninhadas, desconto aplicado em task layer, etc.). No entanto, sua latência de rede no percentil 95 foi **21.08ms**, consideravelmente menor que os **36.93ms** do `/products`.
*   **Suspeita/Achado:** O cenário de testes K6 injeta 10% de erros propositais (422) no `/orders`. Esses erros falham prematuramente nos *guards* lógicos (`require` / `reject`), ou seja, eles **não chegam a bater no banco de dados**. Isso puxa a média estatística geral da rota `/orders` artificialmente para baixo no K6 (já que 1/10 responde de volta instantaneamente na casa de micro-segundos).

### 3.3. Memory Bloat & Pool Retention (CRÍTICO)
No `BASELINE_040`, destacamos especificamente (anotações da rodada 1): *"Memory trend: Flat - no leak"* (ancorada em ~19.5MiB).
Na rodada atual com DB, o consumo iniciou limpo em `5MiB`, engoliu RAM de forma ininterrupta, atingindo `30MiB` na estabilização inicial, e cravando um pico de **61.34MiB** perto do final, sem nunca mais retornar à baseline originária (<20MiB).

**Suspeitas de causa raiz que devem ser checadas na arquitetura do DB (`src/db`):**
1.  **Connection Pool Holding Buffers**: O pool padrão do `sqlx` (ou provider utilziado no framework) está retendo conexões inativas ou os buffers TLS alocados por conexão dentro do Tokio nunca diminuem. Isso resultará num platô de memória "seguro", e não em um leak fatal. Precisamos ver se cai após 1 hora ocioso.
2.  **Prepared Statements Cache Leak**: Se as queries SQL construídas dinamicamente via `pipeline` do MarretaLang (ex: `.where("x", y)`) não estão sendo parametrizadas adequadamente ($1, $2) e sofrem string concatenation, o banco e a cache interna de "prepared statements" do `sqlx` em Rust vão criar um hash entry inédito enchendo a RAM para cada consulta diferente na query string. Essa causa é fatal e causa OOM se não sanada.
3.  **Tokio Async Bloat**: Muitas conexões caindo em Time-wait ou buffers socket do Tokio inchados.

> **Note to Maintainers:** Investigar se o cache de declaração (SQL caching) do worker threads está crescendo linearmente. Fazer um profiler da memória da struct `QueryBuilder` na próxima manutenção do relacional.
