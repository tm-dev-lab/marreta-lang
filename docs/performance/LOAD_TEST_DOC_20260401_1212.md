## 1. Context

First load test of the `doc.*` module (MongoDB) introduced in v0.7.0. The `ecommerce_doc` example
mirrors the `ecommerce` example structure — same three scenarios (health, products, orders) — but
all persistence uses `doc.*` (MongoDB) instead of `db.*` (PostgreSQL).

Key differences from the PostgreSQL run:
- **Schema-less**: no migrations, no `id SERIAL PRIMARY KEY` — `_id` is a MongoDB ObjectId string
- **No transactions**: orders endpoint does not use `transaction {}` (MongoDB transactions require replica sets, out of scope)
- **Same VU profile and thresholds**: health p95 < 200ms, products/orders p95 < 500ms, error rate constraints

## 4. Analysis

### 4.1 MongoDB vs PostgreSQL Latency Comparison

| Scenario | MongoDB p95 | PostgreSQL p95 (pool) | Delta |
|---|---|---|---|
| health | 0.815ms | 0.812ms | equivalent |
| products | 1.582ms | 1.545ms | +0.037ms (+2%) |
| orders | 1.555ms | 1.454ms | +0.101ms (+7%) |

MongoDB and PostgreSQL show **equivalent latency** at p95 under this workload on a local single-node
setup. Both are well within the 500ms threshold with 300–320× headroom. The slight MongoDB
advantage in raw throughput (5,316 vs 5,307 req/s) is within measurement noise.

The orders scenario shows a small p95 increase (+7%) for MongoDB vs PostgreSQL. This is expected:
the PostgreSQL orders endpoint uses a `transaction {}` block (atomic insert + rollback on failure)
while the MongoDB endpoint performs a plain `doc.orders.save`. This also means MongoDB errors are
handled with a simpler code path — less overhead on the success path but no atomic guarantees.

### 4.2 MongoDB Characteristics

- **No schema enforcement**: collections are created on first write. Zero migration overhead.
- **ObjectId as `_id`**: returned as a string, fully usable as a lookup key in subsequent `doc.orders.find(order_id)` calls.
- **No replica-set transactions**: single-node MongoDB does not support multi-document transactions. The ecommerce_doc example uses a validation-first pattern (check product exists, then save order) rather than a transaction block.
- **Schemaless flexibility**: new fields in the payload are stored transparently without schema changes.

### 4.3 Throughput Parity

At 5,316 req/s sustained over 11m30s with 200 VUs, MongoDB delivers **full throughput parity** with
PostgreSQL. The interpreter overhead dominates latency (microseconds), not the database.
