# 025 - Persistence By Convention and Query Navigation

Status: Delivered

## Motivation

An explicit relational-modeling DSL would push Marreta too far toward
relationship configuration, bridge semantics, constraint rules, and
migration-specific knobs. That direction increases cognitive load, makes
schemas feel infrastructural, and pulls the language away from its core
identity.

Marreta is a language for building REST APIs with zero ceremony. The developer
should describe domain entities and business behavior, not manually model
foreign keys, constraint names, or relationship metadata.

This spec restarts the problem from that premise:

- persistence must be activated by a small convention, not a forest of
  configuration knobs;
- relationships must be inferred when they are obvious;
- ambiguity must fail clearly instead of adding syntax to resolve it;
- query navigation must feel like following domain relations, not writing an
  ORM.

## Goals

- Make a schema persistent with one structural cue: `db:`.
- Standardize persistent primary keys as `id`.
- Infer foreign keys from singular references to persistent schemas.
- Infer inverse collection relationships from `list of <persistent schema>`
  only when the inverse is unambiguous.
- Add query navigation for inferred relations.
- Keep schemas readable as domain declarations, not database configuration.
- Keep migrations derived from conventions, not from relational configuration.
- Keep `marreta doctor` project-oriented and usable even in projects with no
  relational database usage.

## Non-goals

- Custom primary key names.
- Composite primary keys.
- Declarative primary-key, generation, cascade, unique, `via`, or bridge
  schema configuration in v1 of this model.
- Many-to-many persistence inference.
- Cascade configuration in schemas.
- Composite or declarative relational uniqueness.
- ORM-style eager loading, joins, or include graphs.
- A second relational configuration DSL layered on top of schemas.

## Design Principle

```text
Schemas describe entities.
Persistence is inferred by convention.
Navigation follows inferred relations.
If the relation is not obvious, Marreta fails instead of asking for more DSL.
```

## Core Idea

Persistent storage is driven by the smallest possible convention set.

Example:

```marreta
schema User
    db: users

    id: integer
    name: string
    orders: list of Order

schema Order
    db: orders

    id: integer
    customer: User
    total: float
```

From this alone, Marreta infers:

- `User` and `Order` are persistent schemas.
- both use `id` as primary key.
- both use database-generated integer ids.
- `Order.customer` generates the foreign key column `customer_id`.
- `User.orders` is the inverse of `Order.customer`.
- query navigation may traverse `Order.customer` and `User.orders`.

The developer models entities. The runtime derives persistence and navigation.

## Required Semantics

These are language/runtime obligations, not recommendations.

1. A schema becomes persistent when it declares `db: <table_name>`.
2. Every persistent schema must declare a field named `id`.
3. In this model, `id` is always the primary key of a persistent schema.
4. The initial supported persistent `id` type is `integer`.
5. In v1, persistent `id: integer` is database-generated by the relational
   backend. Create payloads and inserts are not expected to supply it.
6. The `id` field on a persistent schema must not be optional.
7. Schema names must use PascalCase.
8. A singular field on a persistent schema may reference only another
   persistent schema in v1. Referencing a non-persistent schema is a load-time
   modeling error.
9. This model does not add any explicit relational persistence configuration
   surface to schemas in v1.
10. A singular field whose type is another persistent schema defines a stored
   many-to-one relation.
11. A stored many-to-one relation generates a local foreign key column named
   `<field>_id`.
12. A `list of <persistent schema>` field never generates a local database
   column.
13. A `list of <persistent schema>` field is treated as an inverse navigable
   relation only when the target schema contains exactly one singular
   reference back to the declaring schema.
14. If the target schema contains zero singular references back to the
    declaring schema, project loading fails with a clear modeling error.
15. If the target schema contains more than one singular reference back to
    the declaring schema, project loading fails with a clear ambiguity error.
16. The ambiguity rule applies only to inverse collection inference. Singular
    navigation remains valid because the field is explicit.
17. `list of <non-persistent schema>` retains its existing schema/contract
    meaning and is not treated as a relational navigation.
18. This model does not infer many-to-many persistence structures.
19. Query navigation is allowed only for relations inferred by rules 10 to 15.
20. Fetched relational records expose both the physical foreign key field
    (for example `customer_id`) and the logical relation field (for example
    `customer`) when that relation is inferred by this model.
21. The logical relation field is a navigable relation handle, not an eagerly
    loaded nested record.
22. Navigating a singular relation yields a relation handle that resolves to
    one record when fetched.
23. Navigating an inverse collection relation yields a relation handle that
    resolves to many records when fetched.
24. Singular relation handles support `fetch` and `exists` in v1.
25. Collection relation handles support `fetch`, `exists`, `count`, `where`,
    `order_by`, and `page` in v1.
26. Operations outside those sets are invalid and must fail clearly rather than
    being normalized implicitly.
27. Relation handles compose with the existing query pipeline within those
    operator sets.
28. If a relation cannot be inferred safely, navigation for that relation is
    unavailable and the project must fail at load time instead of guessing.
29. Migrations generated under this model must derive foreign key columns from
    singular references only.
30. Runtime write flows that try to treat a database-generated persistent `id`
    as user-supplied must fail clearly rather than being silently accepted.
31. `marreta migrate *` remains a relational-database workflow. It is only
    relevant when the project defines schemas with `db:`.
32. `marreta doctor` remains a project command, not a relational-database
    utility. It must work for projects with or without `db:` schemas.
33. `marreta doctor` must report persistence/navigation inference from source
    structure alone and must not require a database connection to explain this
    model.
34. The default `marreta doctor` output for this spec must stay domain-oriented
    and convention-oriented. It must not require SQL constraint names or other
    migration internals as primary output.
35. If a project has no `db:` schemas, `marreta doctor` must simply omit or
    skip `db`-persistence reporting; that is not an error.
36. `marreta migrate *` must not derive storage from inverse `list of`
    declarations alone. Storage continues to be derived from singular
    references plus persistent-schema conventions.

## Schema Examples

### Simple one-to-many

```marreta
schema User
    db: users

    id: integer
    name: string
    orders: list of Order

schema Order
    db: orders

    id: integer
    customer: User
    total: float
```

Inferred storage:

- `users.id` is the primary key.
- `orders.id` is the primary key.
- `orders.customer_id` is the foreign key column.

Inferred navigation:

- `Order.customer`
- `User.orders`

### Singular navigation stays explicit

```marreta
schema Order
    db: orders

    id: integer
    customer: User
    billed_by: User
```

This is valid. Both singular relations are explicit:

- `Order.customer`
- `Order.billed_by`

### Ambiguous inverse collection

```marreta
schema User
    db: users

    id: integer
    purchases: list of Order

schema Order
    db: orders

    id: integer
    customer: User
    billed_by: User
```

This is invalid in v1 because `User.purchases` is ambiguous. Marreta must fail
at load time with an error that names the conflicting fields on `Order`
(`customer`, `billed_by`).

### Non-persistent `list of` remains unchanged

```marreta
schema InvoiceItem
    product: string
    quantity: integer

schema InvoicePayload
    items: list of InvoiceItem
```

This is regular schema composition, not persistence/navigation inference.

### Invalid singular reference to non-persistent schema

```marreta
schema InvoicePayload
    total: float

schema Order
    db: orders

    id: integer
    payload: InvoicePayload
```

This is invalid in v1. Persistent schemas may only hold singular references to
other persistent schemas.

## Query Navigation

Navigation must read like following domain relations.

### Singular relation

```marreta
order = db.orders.find(params.id)

# Physical FK field remains visible on the fetched record.
customer_id = order.customer_id

# Logical relation field is navigable.
customer = order.customer >> fetch
```

```marreta
reply 200, {
    id: order.id,
    total: order.total,
    customer: order.customer >> fetch
}
```

### Inverse collection

```marreta
user = db.users.find(params.id)
orders = user.orders >> fetch
```

### Navigation plus pipeline

```marreta
paid_orders = user.orders
    >> where(status: "paid")
    >> order_by("created_at desc")
    >> fetch
```

### Other terminal operations

```marreta
has_orders = user.orders >> exists
count = user.orders >> count
```

## Query Navigation Semantics

1. `record.relation >> fetch` resolves a related record or list of records.
2. Singular navigation fetches at most one related record.
3. Collection navigation fetches zero or more related records.
4. Singular relation handles support only `fetch` and `exists` in v1.
5. Collection relation handles support `fetch`, `exists`, `count`, `where`,
   `order_by`, and `page`.
6. `where`, `order_by`, `page`, and `count` are invalid on singular relation
   handles in v1.
7. The runtime must fail clearly instead of trying to coerce a singular
   relation handle into a collection query.
8. Query navigation must reuse the existing `db.*` runtime rather than
   introducing a second relational execution model.
9. Query navigation must fail clearly when attempted on a field that is not an
   inferred persistent relation.

## Migration Semantics

This spec intentionally keeps migrations narrow.

1. Persistent schemas create tables.
2. `id: integer` maps to the standard generated integer primary key form used
   by Marreta migrations.
3. Singular references to persistent schemas generate `<field>_id` columns and
   foreign key constraints.
4. Singular references to non-persistent schemas are rejected at load time and
   never reach migration generation.
5. `list of <persistent schema>` does not generate columns or bridge tables.
6. This spec does not add relational configuration surface such as cascade or
   explicit uniqueness.

## Migrate

`marreta migrate` is the CLI workflow for relational persistence only.

In this model, `migrate` is responsible for translating persistent schemas into
relational storage derived by convention:

- create tables for schemas that declare `db:`
- create the conventional `id` primary key
- create `<field>_id` foreign key columns for singular persistent references
- create foreign key constraints derived from those singular references

`migrate` is **not** responsible for:

- interpreting `list of <persistent schema>` as storage on its own
- creating bridge tables
- requiring any explicit relational configuration surface
- becoming a general-purpose project command outside relational persistence

If a project has no `db:` schemas, `marreta migrate diff` should report no
relational work to do rather than treating that as an error.

## Load-time Errors

The runtime must reject these cases before the server starts:

- persistent schema without `id`
- persistent schema whose `id` is not `integer`
- persistent schema whose `id` is optional
- any schema syntax attempting to add explicit relational persistence
  configuration in this model
- singular reference from a persistent schema to a non-persistent schema
- `list of <persistent schema>` with no inverse singular reference
- `list of <persistent schema>` with multiple inverse singular references
- navigation attempt over a non-inferred relation
- use of collection-only navigation operators on a singular relation handle

## Doctor

`marreta doctor` should surface this model at the level of convention, not SQL
mechanics.

`doctor` is not a `db` subcommand. It is a project command. This distinction is
important:

- projects may use `doctor` without using relational persistence at all
- projects may use `doctor` before any database is provisioned
- this spec must not turn `doctor` into a relational-only utility

For this model, `doctor` reports how Marreta understands persistence and
navigation from source code. It does not need a live database connection to do
that.

Example:

```text
Persistence (db):
  OK  users persisted by convention
  OK  orders persisted by convention
  OK  orders.customer -> users via customer_id
  OK  users.orders inverse of orders.customer
```

If the project has no `db:` schemas, `doctor` should omit this section or
show an explicit skip such as:

```text
Persistence (db):
  --  no db: schemas declared
```

This is informational, not an error.

This spec does not require doctor to render constraint names, FK naming rules,
or migration-level implementation details as primary output. The focus is the
domain relation understood by the language. If a future verbose mode exposes
SQL details, those details must remain secondary to the project-level summary.

## Implementation Plan

### Phase 1

- Reject unsupported relational persistence configuration in persistent
  schemas.
- Enforce `db:` + `id: integer` as the persistence contract.
- Enforce PascalCase schema names.
- Derive stored foreign keys from singular persistent references.
- Keep `list of <non-persistent>` unchanged.
- Keep `marreta migrate *` scoped to `db:` schemas only.
- Ensure `marreta doctor` remains valid and useful in projects with no
  `db:` schemas.

### Phase 2

- Infer inverse collection relations from unambiguous `list of <persistent>`.
- Add clear load-time errors for zero-match and multi-match inverse cases.
- Expose inferred relations through doctor output.
- Keep doctor output convention-oriented rather than migration-oriented.

### Phase 3

- Add singular relation navigation (`order.customer >> fetch`).
- Add inverse collection navigation (`user.orders >> fetch`).
- Compose relation handles with the existing query pipeline according to their
  allowed operator sets.
- Keep navigation execution inside the existing `db.*` runtime path.

## Test Plan

### Phase 1

- Persistent schema without `id` fails at load time.
- Persistent schema with `id` not typed as `integer` fails at load time.
- Persistent schema with optional `id` fails at load time.
- Persistent `id: integer` is treated as database-generated in create flows.
- Singular persistent reference generates `<field>_id`.
- Singular reference from a persistent schema to a non-persistent schema fails
  at load time.
- `list of <non-persistent>` continues to behave as plain schema composition.
- Non-PascalCase schema names fail with a clear load-time message.
- `marreta migrate diff` with no `db:` schemas reports no relational work
  to do.
- `marreta doctor` in a project with no `db:` schemas still succeeds.

### Phase 2

- `list of <persistent>` resolves when exactly one inverse singular reference
  exists.
- `list of <persistent>` fails when no inverse singular reference exists.
- `list of <persistent>` fails when multiple inverse singular references exist.
- Doctor reports inferred persistence and inferred inverse relations.
- Doctor output stays domain-oriented and does not require SQL internals to be
  useful.

### Phase 3

- Fetched relational records expose both physical FK fields and logical
  navigable relation fields.
- `record.singular_relation >> fetch` returns the related record.
- `record.singular_relation >> exists` works.
- `record.singular_relation >> count` fails clearly.
- `record.singular_relation >> where(...)` fails clearly.
- `record.collection_relation >> fetch` returns the related records.
- `record.collection_relation` composes with `where`, `order_by`, `count`, and
  `exists`.
- Attempting navigation on a non-inferred field fails clearly.

## Validation Plan

This section describes the practical validation expected for this spec. It is
more operational than the phase-oriented `Test Plan` above.

### 1. Migrate validation

This track exists only for projects using relational persistence (`db:`).

Validate with CLI flows such as:

- `marreta migrate diff`
- `marreta migrate generate`
- `marreta migrate apply`
- `marreta migrate rollback`
- `marreta migrate discard`
- `marreta migrate status`
- `marreta migrate list`

Expected checks:

- projects with `db:` schemas generate tables from persistent schemas only
- singular persistent references generate `<field>_id` columns
- inverse `list of <persistent>` does not generate local columns
- projects without `db:` schemas report no relational work to do
- generated migration SQL contains only the storage implied by conventions
- rollback removes or reverts the relational changes introduced by the
  migration being rolled back
- discard removes unapplied migration files cleanly and updates
  `status` / `list` consistently

### 2. Doctor validation

This track applies to any Marreta project, with or without relational
persistence.

Validate with:

- `marreta doctor`
- `marreta doctor --connect` when a database-backed project exists and a
  connection is intentionally part of the test

Expected checks:

- projects without `db:` schemas still succeed in `doctor`
- the `Persistence (db)` section is omitted or explicitly skipped when no
  `db:` schemas exist
- projects with `db:` schemas show convention-oriented persistence inference
- `doctor` explains inferred singular relations and inferred inverse
  collection relations
- `doctor` remains useful without requiring migration internals as primary
  output

### 3. Functional example validation

This spec should be validated with at least one relational example project
whose schemas exercise:

- persistent `db:` schemas
- singular references
- unambiguous inverse `list of <persistent>`
- ambiguous inverse `list of <persistent>` as a failure case
- singular navigation and collection navigation

Recommended validation flow:

1. start the example database
2. run `marreta doctor`
3. run `marreta migrate diff`
4. run `marreta migrate generate`
5. run `marreta migrate apply`
6. run `marreta migrate rollback`
7. verify the schema/state returned to the expected pre-apply form
8. regenerate and validate `marreta migrate discard` on an unapplied migration
9. re-generate and re-apply the migration
10. start `marreta serve`
11. exercise HTTP routes that read and navigate inferred relations

Expected checks:

- writes persist through the inferred FK structure
- reads return the expected records through normal `db.*` operations
- relation navigation returns the same logical data as equivalent manual
  `db.*` queries
- rollback removes the generated tables/constraints/columns expected for that
  migration step
- discard only affects unapplied migrations and leaves applied database state
  untouched
- ambiguous inverse declarations fail before the server starts

### 4. Direct database verification

The relational example must also be verified by opening the database directly
and inspecting the generated schema and stored data.

Recommended checks include SQL against PostgreSQL system catalogs and target
tables, for example:

- list created tables
- inspect table columns in `information_schema.columns`
- inspect foreign keys in `information_schema.table_constraints` and
  `information_schema.key_column_usage`
- inspect inserted rows in the example tables

Expected checks:

- each `db:` schema created the expected table
- each singular persistent reference created the expected `<field>_id` column
- inverse `list of <persistent>` fields created no local columns
- foreign key constraints point to the expected parent table and `id` column
- inserted example data can be read back consistently through both SQL and
  Marreta routes

### 5. Navigation parity validation

For every navigation example introduced by this spec, validate parity between
navigation and explicit query forms.

Examples:

```marreta
customer = order.customer >> fetch
```

must match the same record returned by the explicit relational lookup implied
by the convention.

```marreta
orders = user.orders >> fetch
```

must match the same set returned by the explicit query filtered on the
generated `<field>_id` column.

### 6. Example command expectations

The final delivery should document concrete commands for maintainers to run in
the relational example project, covering:

- doctor without connection
- doctor with connection
- migrate diff/generate/apply/rollback/discard/status/list
- HTTP requests that exercise navigation
- direct `psql` inspection queries for structure and data

## Open Questions

- Should `id` remain explicitly declared in the schema, or can Marreta inject
  it implicitly for persistent schemas in a future version?
- Should navigation support nested fetch chains in v1
  (`order.customer.company >> fetch`) or land only with one hop initially?
- Should doctor surface inferred SQL details behind a verbose flag, while the
  default output stays domain-oriented?

## Deferred

- Many-to-many by convention.
- Any schema syntax for relation disambiguation.
- Custom primary keys.
- Composite keys.
- Cascade behavior configuration.
- Declarative relational uniqueness.
- Eager loading or include graphs.
- Join builder syntax.
- Explicit bridge-table modeling in the language surface.
