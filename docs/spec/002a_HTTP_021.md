## 1. Feature 1 — Multiple take Bindings

### Motivation

Real routes need to read both the request body and headers simultaneously (e.g., JWT auth + payload). The current grammar only allows one `take` per route.

### Syntax

```marreta
route POST "/checkout" take payload, headers
    require headers.authorization else fail 401, "Missing token"
    require payload.cart else fail 400, "Empty cart"
    reply 201, { ok: true }
```

### Design

**`ast.rs`** — `Route.take` becomes `Vec<TakeBinding>` (was `Option<TakeBinding>`):

```rust
Route {
    verb: HttpVerb,
    path: String,
    take: Vec<TakeBinding>,   // was Option<TakeBinding>
    body: Vec<Statement>,
    line: usize,
    column: usize,
}
```

An empty `Vec` means no bindings (replaces `None`). Each binding injects its variable independently.

**`parser.rs`** — `parse_route()` reads a comma-separated list of bindings after `take`:

```
take payload, headers     → [Payload("payload"), Headers("headers")]
take query, payload       → [Query("query"), Payload("payload")]
take payload              → [Payload("payload")]   (single, backward compat)
```

**`server.rs`** — `execute_route()` iterates over `route.take` and injects each binding:

```rust
for binding in &route.take {
    match binding {
        TakeBinding::Payload(name) => { ... }
        TakeBinding::Query(name)   => { ... }
        TakeBinding::Headers(name) => { ... }
        TakeBinding::Form(name)    => { ... }  // Feature 3
        TakeBinding::Raw(name)     => { ... }  // Feature 3
    }
}
```

### Files Changed

| File | Change |
|---|---|
| `src/ast.rs` | `take: Vec<TakeBinding>` |
| `src/parser.rs` | comma-separated take list |
| `src/server.rs` | iterate over Vec<TakeBinding> |
| `src/route_loader.rs` | no change (RouteDefinition mirrors AST) |

### Migration

Existing `take payload` still works — parser produces `vec![TakeBinding::Payload("payload")]`.

### Tests

- Parse: single take, two takes (payload+headers), three takes (payload+query+headers), no take
- Integration: `take payload, headers` route — auth header + body simultaneously

## 2. Feature 2 — `env` Object (Environment Variables)

### Motivation

Secrets (DB passwords, JWT keys, Stripe API keys) must not be hardcoded. The server should expose OS environment variables as a read-only `env` map accessible in any route.

### Syntax

```marreta
route POST "/charge"
    key = env.STRIPE_KEY
    require key else fail 500, "stripe key not configured"
    reply 200, { charged: true }
```

### Design

**`config.rs`** — extend `MarretaConfig::load()` to also load from a `.env` file using the `dotenvy` crate (reads `marreta.env` automatically before `std::env::vars()`).

**`main.rs`** — after startup stmts are executed, inject `env` into the environment:

```rust
// Collect all env vars into a Value::Map
let env_map: HashMap<String, Value> = std::env::vars()
    .map(|(k, v)| (k, Value::String(v)))
    .collect();
interp.env_set("env".to_string(), Value::Map(Arc::new(RwLock::new(env_map))));
```

This must happen **before** `Arc::new(interp.into_environment())` so that `env` is part of the shared (read-only) startup state cloned per request.

**No new syntax or AST nodes required** — `env` is just a regular `Value::Map` variable in the global scope. `env.STRIPE_KEY` is a standard property access.

### New dependency

```toml
dotenvy = "0.15"
```

`dotenvy::dotenv_override()` loads `.env` (if present) into `std::env` before the server starts. Then `std::env::vars()` gives the full merged set.

### Files Changed

| File | Change |
|---|---|
| `Cargo.toml` | add `dotenvy = "0.15"` |
| `src/config.rs` | call `dotenvy::dotenv_override()` in `load()` |
| `src/main.rs` | inject `env` map after startup stmts |

### Tests

- Unit: `env` variable is a Map in the environment after `run_serve` setup
- Integration: route reads `env.TEST_KEY` (set via `std::env::set_var` in test setup)

## 3. Feature 3 — Dynamic Request Types (`form` / `raw`)

### Motivation

- `take form` — for HTML form submissions (`application/x-www-form-urlencoded`)
- `take raw` — for webhooks with raw body + HMAC signature verification (`application/octet-stream` or `text/plain`)

### Syntax

```marreta
# Form upload — receives a Value::Map with string values
route POST "/contact" take form
    require form.email else fail 422, "email required"
    reply 200, { received: true }

# Raw webhook — receives body as Value::String
route POST "/webhook" take raw
    require raw else fail 400, "empty body"
    reply 200, { ok: true }
```

### Design

**`ast.rs`** — two new `TakeBinding` variants:

```rust
pub enum TakeBinding {
    Payload(String),   // JSON body → Value::Map / Value::*
    Query(String),     // query string → Value::Map<String, String>
    Headers(String),   // headers → Value::Map<String, String>
    Form(String),      // form body → Value::Map<String, String>
    Raw(String),       // raw body bytes → Value::String
}
```

**`parser.rs`** — `parse_take_binding()` recognises `form` and `raw` keywords:

```
"query"   → TakeBinding::Query(name)
"headers" → TakeBinding::Headers(name)
"form"    → TakeBinding::Form(name)
"raw"     → TakeBinding::Raw(name)
_         → TakeBinding::Payload(name)  (default)
```

> Note: `form` and `raw` are parsed as identifiers (not reserved words), matching the existing pattern for `query`/`headers`.

**`server.rs`** — handler signature must support raw bytes extraction alongside JSON. Use axum's `Bytes` extractor for `Raw` and `Form` extractor for `Form`:

```rust
// Handler receives:
body_bytes: Option<Bytes>,
body_json: Option<Json<JsonValue>>,
form_data: Option<Form<HashMap<String, String>>>,
```

Alternatively (simpler): receive `body_bytes: Bytes` always and parse internally based on binding type.

**Chosen approach:** receive `body_bytes: axum::body::Bytes` (always available), and based on binding type:
- `Payload` → parse as JSON via `serde_json::from_slice`
- `Form` → parse as form-encoded via `serde_urlencoded`
- `Raw` → wrap bytes as `Value::String` (UTF-8, lossy)

### New dependency

```toml
serde_urlencoded = "0.7"
```

### Files Changed

| File | Change |
|---|---|
| `src/ast.rs` | `TakeBinding::Form`, `TakeBinding::Raw` |
| `src/parser.rs` | recognise `form`, `raw` keywords in take |
| `src/server.rs` | switch from `Option<Json<JsonValue>>` to `Bytes`; dispatch by binding type |
| `Cargo.toml` | add `serde_urlencoded = "0.7"` |

### Tests

- Parse: `take form`, `take raw`, `take payload, form` (multi-binding)
- Integration: POST form data → `form.field` accessible; POST raw bytes → `raw` is a string

## 4. Feature 4 — Response Modifiers (`html` / `text` / redirect)

### Motivation

Not all APIs return JSON. HTML error pages, CSV exports, webhook confirmations, and redirects are common real-world requirements.

### Syntax

```marreta
# HTML response
reply html 200, "<h1>Internal Error</h1>"

# Plain text response
reply text 201, "Webhook received successfully"

# Redirect (302) with Location header
reply 302, null, { Location: "https://mysite.com/login" }

# Default (current behaviour) — JSON
reply 200, { ok: true }
```

### Grammar

```
Reply ::= "reply" [ContentType] INTEGER "," Expression ["," MapLiteral]

ContentType ::= "html" | "text"
```

The optional third argument `{ ... }` is a **headers map** passed as an extra response header set. This enables redirects and custom headers without a dedicated `redirect` keyword.

### AST

**`ast.rs`** — extend `Statement::Reply`:

```rust
Statement::Reply {
    status_code: u64,
    content_type: ReplyContentType,   // NEW
    body: Expression,
    headers: Option<Expression>,      // NEW — optional headers map
    line: usize,
    column: usize,
}

pub enum ReplyContentType {
    Json,   // default
    Html,
    Text,
}
```

### Interpreter

**`interpreter.rs`** — `Statement::Reply` serializes body based on `content_type`:

- `Json` → `value_to_json(&body).to_string()` (current behaviour)
- `Html` | `Text` → `body.to_string()` (no JSON wrapping)

**`error.rs`** — extend `MarretaError::HttpResponse`:

```rust
HttpResponse {
    status_code: u16,
    body: String,
    content_type: String,   // "application/json", "text/html", "text/plain"
    extra_headers: Vec<(String, String)>,  // NEW — for Location etc.
    is_error: bool,
}
```

### Server

**`server.rs`** — `execute_route()` builds response with correct `Content-Type` header and optional extra headers:

```rust
let mut builder = Response::builder().status(status);
builder = builder.header("Content-Type", content_type);
for (k, v) in extra_headers {
    builder = builder.header(k, v);
}
builder.body(body).into_response()
```

### Files Changed

| File | Change |
|---|---|
| `src/ast.rs` | `ReplyContentType` enum, extend `Statement::Reply` |
| `src/parser.rs` | optional `html`/`text` after `reply`, optional 3rd arg |
| `src/interpreter.rs` | serialize by content type, propagate `content_type` + `extra_headers` |
| `src/error.rs` | extend `HttpResponse` with `content_type` + `extra_headers` |
| `src/server.rs` | build response with correct headers |

### Tests

- Parse: `reply html 200, "..."`, `reply text 201, "..."`, `reply 302, null, { Location: "..." }`
- Interpreter: `reply html` emits `HttpResponse` with `content_type: "text/html"`
- Integration: HTML route returns `Content-Type: text/html`; redirect route returns `302 Location` header

## 5. Feature 5 — CORS Middleware

### Motivation

Without CORS headers, browsers block cross-origin requests from frontend apps (React, Vue, etc.) running on different ports/domains.

### Design

**`Cargo.toml`** — add `tower-http` with `cors` feature:

```toml
tower-http = { version = "0.6", features = ["cors"] }
```

**`server.rs`** — apply `CorsLayer` to the router:

```rust
use tower_http::cors::{Any, CorsLayer};

let cors = CorsLayer::new()
    .allow_origin(Any)
    .allow_methods(Any)
    .allow_headers(Any);

let app = router.layer(cors);
axum::serve(listener, app).await?;
```

**`config.rs`** — add `cors_enabled: bool` and `cors_origin: String` to `MarretaConfig`:

```rust
pub struct MarretaConfig {
    pub host: String,
    pub port: u16,
    pub cors_enabled: bool,    // NEW — default: true
    pub cors_origin: String,   // NEW — default: "*"
}
```

Controlled via `marreta.env`:

```env
MARRETA_CORS=true
MARRETA_CORS_ORIGIN=*
```

### Files Changed

| File | Change |
|---|---|
| `Cargo.toml` | add `tower-http = { version = "0.6", features = ["cors"] }` |
| `src/config.rs` | `cors_enabled`, `cors_origin` fields |
| `src/server.rs` | apply `CorsLayer` conditionally based on config |
| `src/main.rs` | pass config to `serve()` |

### Tests

- Unit: `MarretaConfig` reads `MARRETA_CORS` and `MARRETA_CORS_ORIGIN`
- Integration: OPTIONS preflight request returns `Access-Control-Allow-Origin: *`

## 6. Feature 6 — 204 No Content Body Fix

### Motivation

RFC 9110 §15.3.5 states that `204 No Content` responses **MUST NOT** include a message body. Currently, `reply 204, null` serializes `null` as a 4-byte JSON string in the response body.

### Current behaviour

```rust
// server.rs — execute_route
Err(MarretaError::HttpResponse { status_code, body_json, .. }) => {
    let json = serde_json::from_str(&body_json).unwrap_or(JsonValue::Null);
    (status, Json(json)).into_response()  // ← sends "null" even for 204
}
```

### Fix

**`server.rs`** — intercept 204 and any response where body is `null`, forcing an empty body:

```rust
Err(MarretaError::HttpResponse { status_code, body_json, .. }) => {
    let status = StatusCode::from_u16(status_code)
        .unwrap_or(StatusCode::INTERNAL_SERVER_ERROR);

    // 204 and 304 must have no body per RFC 9110
    if status == StatusCode::NO_CONTENT || status == StatusCode::NOT_MODIFIED {
        return status.into_response();
    }

    // Null body → empty 200 (or whatever status)
    let json: JsonValue = serde_json::from_str(&body_json).unwrap_or(JsonValue::Null);
    if json.is_null() && status == StatusCode::OK {
        return (status, Json(json)).into_response(); // caller may want explicit null
    }

    (status, Json(json)).into_response()
}
```

More precisely: if `status_code == 204`, return `StatusCode::NO_CONTENT.into_response()` with **no body**, regardless of what was passed to `reply`.

### Files Changed

| File | Change |
|---|---|
| `src/server.rs` | guard in `execute_route` for 204/304 — return empty body |

### Tests

- Integration: `reply 204, null` → response body length is 0
- Integration: `reply 200, null` → body is `null` (4 bytes, valid JSON)

## 7. Implementation Order

Features are ordered by impact/dependency:

| # | Feature | Effort | Deps |
|---|---|---|---|
| 1 | Feature 6 — 204 fix | XS | none |
| 2 | Feature 5 — CORS | S | `tower-http` |
| 3 | Feature 2 — `env` object | S | `dotenvy` |
| 4 | Feature 1 — Multiple `take` | M | none |
| 5 | Feature 3 — `form`/`raw` types | M | Feature 1, `serde_urlencoded` |
| 6 | Feature 4 — Response modifiers | L | AST + parser + interpreter + server |

Feature 4 is the most invasive (touches 4 files including `error.rs` and the `HttpResponse` control-flow variant). It must come last.

## Known Constraints

- **Feature 4** extends `MarretaError::HttpResponse` with new fields. The `body_json: String` approach (for `PartialEq + Clone`) is kept; new fields `content_type: String` and `extra_headers: Vec<(String, String)>` are also `Clone + PartialEq`-safe.
- **Feature 1** is a breaking change to `Route.take` type (`Option` → `Vec`). All existing tests using `take: Some(...)` / `take: None` must be updated to `take: vec![...]` / `take: vec![]`.
- **Feature 3** requires changing the axum handler signature from `Option<Json<JsonValue>>` to `Bytes` for the body extractor. This is backward-compatible at the HTTP level.
