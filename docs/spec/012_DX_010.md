# MarretaLang — Implementation Plan: Developer Experience (DX) & Tooling

> Status: Partially delivered.
> Delivered: VS Code TextMate grammar, language configuration, and packaged
> `.vsix` bundles under `docs/vscode-marreta`.
> Pending: LSP, formatter, linter, and project bootstrap remain tracked by the
> public-v1 tooling roadmap.

## Context and Design Rationale

A programming language with a beautiful architecture but terrible tooling is an orphaned project. The Developer Experience (DX) defines adoption. Modern developers expect syntax highlighting out-of-the-box and deeply resent compiling code only to discover a typo.

### Why a VS Code Extension instead of a Custom IDE?
Building an entire Integrated Development Environment (IDE) from scratch — encompassing rendering, window management, plugin systems, and a text engine — requires a dedicated team working for years. It is an anti-pattern for a single DSL.

The industry standard (adopted by Rust, Go, Python) is to integrate the language directly into the editor the market already uses: **Visual Studio Code**.

### Why split into Phase 1 (Highlighting) and Phase 2 (LSP)?
- **Syntax Highlighting (TextMate)** is a lightweight, purely declarative effort using Regex. It provides an immediate injection of DX comfort and makes the code vastly more readable during the core engine implementation phases.
- **Language Server Protocol (LSP)** requires a new Rust binary (`marreta-ls`), an architectural refactor to make the AST parser fault-tolerant (recovering from incomplete lines), and JSON-RPC communication. The LSP is planned for a later phase (v0.10.0), after the language's core abstractions (SQL, Caches, Modules) have fully stabilized.

## Scope

### Phase 1: High-Fidelity Syntax Highlighting (Low Complexity)

Create a scaffolded VS Code extension (`vscode-marreta`) focusing entirely on aesthetics and basic token recognition.

- **Theme & Icons:** Register the `.marreta` file extension with VS Code and assign the "Martim" logo as the official file icon.
- **TextMate Grammar (`marreta.tmLanguage.json`):** Map the language's minimal reserved words to standard TextMate scopes to ensure they colorize correctly regardless of the user's VS Code theme (Dark+, Dracula, standard Light, etc.).
  - `keyword.control`: `task`, `route`, `if`, `else`, `match`, `fallback`, `require`, `reject`
  - `keyword.operator`: `>>`, `*>>`, `->`, `:`, `<`, `>`, `=`, `like`, `in`, `and`, `or`, `not`
  - `storage.type`: `schema`, `string`, `integer`, `float`, `boolean`, `list`, `map`, `of`
  - `support.function.builtin`: `db`, `doc`, `cache`, `queue`
  - `entity.name.function`: Identifier names that follow `task`
- **Indentation Rules (`language-configuration.json`):** Define standard indentation rules. Pressing `Enter` after a `route`, `task`, or `match` block should automatically increase the indent level by 1 tab (or 4 spaces).
- **Comment Toggling:** Configure `#` as the official line-comment character so `Ctrl+/` works natively.

### Phase 2: The Language Server Protocol / LSP (High Complexity)

Create the architectural brain of the extension. The LSP runs the Rust Engine AST in the background while the user types.

- **`marreta-ls` Binary:** A new crate in the Marreta workspace. It speaks standard JSON-RPC over `stdio` to the VS Code extension client.
- **Diagnostics (Live Error Reporting):** Re-route `parser.rs` output to the LSP. Because the Pratt Parser is hand-rolled, it knows the exact line/column of an error. The LSP will translate these to red squiggly lines in the editor (`textDocument/publishDiagnostics`).
- **Semantic Tokens (Advanced Highlighting):** Beyond Regex, the LSP reads the AST and colors symbols based on runtime awareness (e.g., coloring a variable bound `as order_payload` slightly differently than an unbound argument).
- **Hover Information (`textDocument/hover`):** When the developer hovers over `payload` defined as `schema address`, the LSP responds with a tooltip showing the properties (`street: string`, `city: string`).
- **Completions (`textDocument/completion`):** 
  - Standard variables in scope.
  - Native namespace chaining (`db.` → `table` → `save`, `find`, `find_all`).
- **Auto-Formatting (`textDocument/formatting`):** A canonical code formatter (like `gofmt` or `rustfmt`) that resolves spacing inside pipelines and normalizes indentation.

## Architecture (Phase 2)

```text
marreta-lang/
├── src/            # Core Interpreter & Parser
├── examples/       # End-to-End Projects
├── marreta-ls/     # The Language Server Crate (Rust, uses `tower-lsp`)
│   ├── main.rs     # JSON-RPC server loop
│   └── backend.rs  # Maps AST data to LSP Types
└── vscode-marreta/ # The VS Code Extension (TypeScript)
    ├── package.json
    ├── syntaxes/
    │   └── marreta.tmLanguage.json
    └── src/
        └── extension.ts # Boots marreta-ls binary and connects client
```

*Note: The Marreta Parser must gradually adapt to "Error Recovery" during Phase 2. A standard parser dies on the first syntax error and stops reading. For an LSP, the AST must produce partial trees so that autocomplete on line 20 still works even if line 5 contains a missing comma.*

## Acceptance Criteria

### Phase 1 — Syntax Highlighting
- [ ] VS Code recognizes `.marreta` files and assigns them an icon.
- [ ] The `route`, `task`, `require`, and `schema` reserved words display correctly under the `keyword` token scope.
- [ ] `ctrl+/` (Toggle Comment) places and removes a `#` natively.
- [ ] VS Code automatically indents the next line after pressing `Enter` on a `task x` declaration line.
- [ ] Pipeline operators (`>>`, `*>>`) have correct operator highlighting.

### Phase 2 — The Language Server Protocol (LSP)
- [ ] `marreta-ls` binary compiles successfully and understands standard LSP `initialize` commands.
- [ ] The VS Code extension bootstraps the `marreta-ls` binary when opening a project implicitly.
- [ ] Extraneous characters or grammar violations output red squiggly lines via `Diagnostics` as the user types, using coordinate mapping from `parser.rs`.
- [ ] Hovering over a `schema` block renders its fully parsed structure in a tooltip.
- [ ] Hovering over `db.users` identifies it as a generic `QueryBuilder` pipeline opener.

---

## Implementation Rationale

1. We are explicitly decoupling Editor Aesthetics from Programmatic Intelligence. Phase 1 provides 80% of the perceived DX value with 2% of the effort.
2. The LSP will exclusively use `tower-lsp`. Rust provides world-class tooling for building language servers without needing to rewrite the JSON-RPC spec from scratch.
3. The design deliberately treats `marreta-ls` as a sub-crate within the same mono-repo. This guarantees the AST models (`Expr`, `Statement`) are always perfectly in sync with the compiler.
