# MarretaLang — API Contracts & Auto-Doc (v0.3.1)

> Status: Delivered.

> **Meta:** Eliminate boilerplate payload validation through rigid *Schemas* declared at the top of the file. Use these definitions to guarantee the HTTP contract and manually auto-generate interactive documentation (OpenAPI/Swagger) within the engine itself, without extra effort from the developer.

---

## 1. Syntax Design

### 1.1 `schema` Declaration
A new Top-Level instruction called `schema`, defined outside of routes. It supports primitive types in **lowercase** and the `?` flag for optional fields.

> **Note (tokens):** Types use lowercase keywords (`string`, `integer`, `float`, `boolean`, `list`, `map`) to avoid conflicts with the existing `TokenKind::StringLiteral` in the lexer.

```marreta
schema user_payload
    name: string
    age: integer
    email?: string     # Optional — `?` attached to the field name (Ruby-style, supported by lexer)
    is_active: boolean
```

### 1.2 "Type Binding" in Route
The route binds the Schema via a dedicated `as <SchemaName>` field — **without altering** `TakeBinding`.

```marreta
route POST "/users" take payload as user_payload
    # If Axios sends JSON missing "name" or with "age" as a String,
    # Axum AUTOMATICALLY returns a 422 Unprocessable Entity.
    # From here downwards, payload.name is guaranteed to be a string.

    db.users.save(payload)
    reply 201, { id: 1 }
```

## 2. Configuration via Environment Variables (`env`)

The documentation page feature will be native, but highly controllable without compromising security in production natively.

The `config.rs` framework from V0.2.1 will accept new injectable fields via `marreta.env`:

```env
# Enables or disables the compilation and display of the documentation screen (default: true)
MARRETA_DOCS_ENABLED=true

# Changes the route where the engine serves the HTML for Swagger UI (default: /docs)
MARRETA_DOCS_PATH=/docs
```

## 3. Implementation Under The Hood (Rust Engine)

### Phase 1: Lexer & AST

**`token.rs`**: Add `Schema`, `As` tokens and lowercase types: `TokenKind::TypeString`, `TokenKind::TypeInteger`, `TokenKind::TypeFloat`, `TokenKind::TypeBoolean`, `TokenKind::TypeList`, `TokenKind::TypeMap`.

> **Why lowercase?** Tokens like `String`, `Integer`, etc. would clash with `TokenKind::StringLiteral` and numeric literals. Using `string`, `integer`, etc., as dedicated keywords keeps the lexer unambiguous.

**`ast.rs`**:
- New node variant `Statement::Schema { name: String, fields: HashMap<String, SchemaField> }`.
- `SchemaField` carries `type_name: SchemaType` and `optional: bool`.
- `SchemaType` enum: `StringType | IntegerType | FloatType | BooleanType | ListType | MapType`.
- **`Statement::Route` gains field `schema: Option<String>`** — points to the schema name bound via `as`. The `TakeBinding` **does not change** (prevents breaking the recent `Option` → `Vec` migration made in v0.2.1).

> **Regarding `?` (optional fields):** The `?` is placed on the field **name** (`email?:`), taking advantage of the lexer's existing support for Ruby-style identifiers (`empty?`). The parser extracts the `?` from the name and sets `optional: true` on the `SchemaField`. No lexer extension required.

**`route_loader.rs`**:
- `RouteRegistry` gains a third field: `pub schemas: HashMap<String, SchemaDefinition>`.
- `SchemaDefinition` is equivalent to the AST fields map, but in a resolved/cloned form for runtime lookup.
- The loader populates `schemas` upon finding `Statement::Schema` in the program.

### Phase 2: The Type Checker Validator
Create a `validator.rs` module.
During `execute_route()`, right after injecting the `payload` into memory (extracted via JSON bytes):
1. The Engine checks if the route defined `schema: Some("user_payload")`.
2. If so, it looks up `user_payload` in `RouteRegistry.schemas`.
3. It iterates over the `Value::Map` validating if:
   - Mandatory fields exist.
   - The native `Value::*` of each key is consistent with the `SchemaType`.
4. In case of a violation, it generates an explicit `MarretaError::HttpResponse` returning `HTTP 422` with a technical array: `{"error": "field 'age' expected integer, got string"}`. The Interpreter aborts the execution gracefully.

### Phase 3: OpenAPI 3.0 Generation (The Documentation Engine)
Create an `openapi.rs` module.
It is not necessary to use complex dependencies (like the `utoipa` crate). Just use `serde_json::json!` to generate the object.

Happens at startup (`marreta serve`):
1. The Engine reads the entire `RouteRegistry` tree.
2. For each defined route (`GET /info`, `POST /users`):
   - Adds the PATH and VERB into the OpenAPI *paths*.
   - Reads the schemas bound via `schema: Some(X)` and injects validation into *components/schemas*.
3. The final result of this scan is kept permanently in RAM in an `Arc<String>` object (the Swagger JSON).

### Phase 4: Swagger UI on Axum
If the `MARRETA_DOCS_ENABLED` control is true:
In `server.rs`, before binding the API, Axum injects two transparent routes:

1. **`GET /openapi.json`**: Returns the compiled JSON model held in the Memory Stack (Phase 3).
2. **`GET /docs` (Or path configured via PATH)**: Returns raw static HTML containing the official Swagger-UI CDN script tags pointing to the native `/openapi.json`.

```rust
// server.rs pseudo-code
if config.docs_enabled {
    let docs_path = config.docs_path.clone();

    app = app.route("/openapi.json", get(serve_openapi_json))
             .route(&docs_path, get(serve_swagger_ui_html));
}
```

> **CDN Caveat:** The Swagger UI HTML uses scripts via CDN (`unpkg.com`). In offline or air-gapped environments, the UI won't load. Recommended strategy: use `include_str!` to embed a minimal hardcoded HTML with the CDN links — and document this limitation. Bundling Swagger UI static assets via `include_bytes!` is possible but increases the binary size; left as a future evolution (v0.4.x).

## 4. Acceptance Criteria

1. **(AST & Engine)** Lexer, Tokens, and Parser can extract `schema X\n  field: type` (with lowercase types and `field?:` for optionals) and `route ... as X`.
2. **(RouteRegistry)** `schemas` populated correctly at startup; `Statement::Route.schema` points to the correct name.
3. **(Validator)** Sending a `POST /users` with the wrong format returns a `422` status with `json({ "error": "field 'age' expected integer, got string" })`.
4. **(OpenAPI Builder)** The `openapi.rs` engine transforms the tree into the Swagger `.json` format.
5. **(Config)** `marreta.env` processes the `MARRETA_DOCS_ENABLED` and `MARRETA_DOCS_PATH` keys.
6. **(Axum)** By opening the browser at `localhost:8080/docs`, the user sees all designed routes of their API in interactive format, with the *Try It Out* button showing payload inputs already assembled based on the Schema.
